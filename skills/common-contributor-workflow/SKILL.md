# Common Contributor Workflow (CanvasCreator)

Use this workflow for routine contributions: bug fixes, small features, docs updates, test updates, and build/release hygiene.

## When to use this skill

Trigger this skill when the user asks to:
- fix or add behavior in CanvasCreator,
- update docs/changelog/release notes,
- add or adjust tests,
- prepare or review a pull request.

## Repository quick map

- `src/` — main library logic (core behavior)
- `styles/` + `canvascreator.min.css` — editable CSS source and minified output
- `tests/` — Jest unit tests
- `scripts/` — utility scripts (export, notes, version updates)
- `dist/` — build artifacts (generated by build)

## Standard workflow

1. **Clarify scope first**
   - Identify exactly what should change and what should stay untouched.
   - Prefer minimal, focused edits.

2. **Inspect before editing**
   - Read only relevant files.
   - Reuse existing helpers/patterns from `src/helpers.js`, `src/main.js`, and tests.

3. **Implement in source files**
   - Make changes in canonical sources (`src/`, `styles/`, `scripts/`, `README.md`, etc.).
   - Do not manually edit generated `dist/` unless task explicitly requires built artifacts.

4. **Update tests with behavior changes**
   - Add/adjust Jest tests in `tests/` for any logic change.
   - Keep test names behavior-focused and deterministic.

5. **Run validation commands**
   - Preferred sequence:
     - `npm test`
     - `npm run build` (if code, style, or packaging changed)
   - If a command fails due to environment limitations, report clearly.

6. **Docs and release notes hygiene**
   - If user-facing behavior changed, update `README.md` and/or `CHANGELOG.md`.
   - Keep entries concise and specific.

7. **Prepare a clean PR**
   - Verify git diff is scoped to the requested change.
   - Write a PR title in imperative style.
   - In PR body include: summary, tests run, and any follow-ups.

## Conventions

- Keep functions small and reuse existing utilities where possible.
- Preserve existing API shape unless the task explicitly requests a breaking change.
- Avoid unrelated formatting churn.
- For CSS changes, edit `styles/canvascreator.css`; regenerate minified CSS only when needed by the task.

## Fast command reference

```bash
npm install
npm test
npm run build
npm run export -- --locale en --format svg --canvas apiBusinessModelCanvas
```

## Completion checklist

- [ ] Change is limited to requested scope.
- [ ] Tests added/updated when behavior changed.
- [ ] Validation commands run (or limitations documented).
- [ ] Relevant docs/changelog updated.
- [ ] Commit message is clear and specific.
